预言机返回了数值，或者请求成功，都不表示一条市场更新已经完整结束。运营方还必须能够说明：这次更新使用了哪套策略、哪些证据被采纳或排除、谁认证了决策、更新前是什么状态，以及新状态为什么可以最终生效。

我们的市场操作回执（`Market Operation Receipt v0`）原型聚焦于检验一个更具体的判断：**市场运营中真正有用的基本单元，不是观测结果或证书本身，而是一条把证据、权限和最终状态转换绑定在一起的唯一终态回执。** Lyquor 原型清晰表达了这条边界。传统单一运营方协调服务能以更低成本守住相同的业务规则，却无法提供同样的多方授权边界。

```text
本地观测
   |
   v
版本化策略 -> 认证决策 -> 有序状态转换
                            |
                            v
                      每轮唯一终态回执
```

## 回执不只是日志

[上一篇市场运营分析](/zh-Hans/blog/2026-07-14-tradexyz-lyquor-market-operations-layer)留下了一个问题：一条市场更新至少应携带哪些证据和备用策略，才能影响风险或结算？本次原型把这个问题压缩成一个可执行状态机，只处理一次 `BTC/USD` 指数价格更新。

“回执”之所以重要，是因为普通日志只是某个进程留下的观测。日志可以解释这个进程声称自己做了什么，却不一定约束它有权修改哪些状态。原型则把回执设计成终态转换的规范化输出。

每轮操作都会记录适用策略、变更前状态、采用的证据、参与授权的主体以及最终结果。终态回执需要回答：

| 问题 | 回执中的证据 |
| --- | --- |
| 决定了哪项操作？ | 操作标识与终态 |
| 使用了哪套规则？ | 策略标识与版本 |
| 状态发生了什么变化？ | 决策前后的状态 |
| 哪些输入被采纳？ | 被采纳的证据 |
| 哪些输入未被采纳，为什么？ | 被排除的证据与原因 |
| 谁授权了这次操作？ | 参与认证的主体 |
| 为什么进入终态？ | 接受、拒绝、过期或取消的原因 |

这些终态必须分开。“拒绝”表示系统已经获得足够的签名证据，可以确认业务规则不允许接受本次更新；“过期”表示系统在形成接受或拒绝决策之前遇到活性失败，因此由受权运营方关闭操作，并保持价格不变。如果把两种情况合并，就无法区分“数据不合格”和“委员会不可用”。

## 我们验证了什么

原型运行在本地多节点环境中，使用可控数据源，只模拟一次指数价格更新。业务规则要求多个节点形成门限，并且证据来自不止一个独立数据源。原型明确不接入真实市场、真实资金或公共网络。

每个节点可以得到不同的外部观测结果。应用策略判断现有证据是否充分，只有经过认证的决策才能改变共享状态，而最终状态只保留一条对应本次操作的回执。

这与 Lyquor 文档描述的边界一致：[实例函数（`instance function`）可以执行本地计算和外部 I/O，网络函数（`network function`）则确定性地更新共享状态](https://docs.lyquor.dev/docs/ldk/)。[两阶段预言机流程（`two-phase oracle flow`）会采集各节点输入、聚合结果，并授权一次认证网络调用（`certified network call`）](https://docs.lyquor.dev/docs/ldk/oracles/)。框架处理协议签名、证书聚合和目标绑定，应用仍要负责自己的业务策略。

判定逻辑检查了不同输入顺序和合法门限组合。这个检查很关键：异步输入到达顺序可能把“取第一个门限结果”变成没有明说的非确定性来源。

故障矩阵覆盖正常更新、离群值、数据源不可用或过期、输入格式错误、价格跳变过大、业务门限不足、节点不可用、决策重放、操作期间更新策略、提议节点切换、节点恢复和连续多轮操作。

其中三个结果最有价值：

- 重放同一个已认证决策，不会产生第二次状态变化或第二条回执；
- 旧操作的决策不能在后续操作完成后修改状态；
- 节点恢复后可以追平已接受的状态，而且不会重复生成回执。

验证不把请求成功返回当作通过条件，而是检查最终状态和回执：终态必须一致，失败决策不能改变原值，每次操作不能生成重复回执。

## 协调服务也通过了——这正是重点

为了避免把 Lyquor 与一个故意做弱的对照组比较，我们实现了一个由单一运营方控制的传统协调服务。它负责收集观测、执行相同的业务规则、决定是否更新状态并保存回执。下文简称“协调服务”。

协调服务同样守住了预期业务不变量，包括重放保护、策略变更、恢复和连续操作。这是一个重要的负向结果：**写出正确的验证逻辑，并不要求分布式认证。**

真正的差异出现在权限边界：

| 边界 | Lyquor 原型 | 单一运营方协调服务 |
| --- | --- | --- |
| 业务规则 | 确定性验证策略 | 相同的验证策略 |
| 最终生效权限 | 认证更新写入共享状态 | 一个受信任写入方 |
| 重放保护 | 依据共享终态强制执行 | 由协调服务及其存储保证 |
| 恢复方式 | 多节点收敛到共享状态 | 进程与存储恢复 |
| 审计证据 | 回执与多方授权证据 | 回执与运营方控制的记录 |
| 状态修改边界 | 单一运营方不能独自生成认证结果 | 受信任写入方最终控制记录 |

在这个有限意义上，Lyquor 更透明：谁参与了授权、哪项决策最终生效，以及各方是否看到同一份共享状态，都更容易被独立验证。但这不表示底层价格一定更真实、业务策略一定更正确，或系统已经不需要信任。如果多个节点依赖同一个错误数据源，或者共同执行了一套错误策略，它们仍然可能认证错误结果。信任只是从单一写入方转移到参与方、策略和证据链，并没有消失。只有当参与方具有实际独立性、策略可检查、证据可追溯时，这种可信度提升才成立。

在本地环境中，协调服务的运行和维护明显更轻。这些本地耗时不适合作为生产性能指标，但成本方向很清楚：Lyquor 在这里的价值不在执行速度，而在更强的权限和证据边界。

如果任务只属于一个团队，且单一写入方本来就是预期的权威来源，传统协调服务是更直接的设计。现有存储权限、签名日志或只追加审计服务可能已经足够。只有当多个参与方必须共同同意状态变化，或者不能允许单一运营方静默改写已接受结果时，Lyquor 路径才开始体现价值。

## 原型没有证明什么

原型有条件地支持了技术假设：Lyquor 能够表达存在差异的本地观测、认证决策、唯一终态回执、重放保护和多节点恢复；它没有证明测试中的价格策略在经济上合理。

原型也没有测试公网延迟和成本、真实交易所 API、数据许可、恶意运营方、生产密钥管理、抗拒绝服务能力、监控，或客户是否愿意采用这套流程。当门限无法形成时，系统仍需要受权的过期处理路径；认证机制不会替代明确的活性与恢复策略。

因此，结论不是“Lyquor 是更好的预言机”，而是：

```text
如果单一受信任写入方本来就是预期权威，使用传统协调服务。
如果权限本身需要受到约束，且终态必须携带多方证据，
再考虑经过认证的共享状态设计。
```

因此，真正的问题不是还可以增加哪些市场功能，而是：**哪些现实的市场工作流，需要多个参与方共同验证一次状态变化，同时又不能把可信记录完全交给单一运营方控制？** 只有当这种需求确实存在，引入额外的系统复杂度才可能是合理的。
